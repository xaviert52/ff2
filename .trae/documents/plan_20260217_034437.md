# Plan de Implementación: Sistema de Gestión de Flujos Extendido

Basado en la arquitectura actual (Servicio `RA` en Node.js para gestión y Servicio `Worker` en Go para ejecución), propongo el siguiente plan de evolución para transformar el sistema en una plataforma de orquestación empresarial.

## Fase 1: Gestión de Conectores (Servicio RA)
El primer paso es establecer la "fuente de la verdad" para los conectores y sus configuraciones.

### 1.1 Modelado de Datos (MySQL)
Crear migraciones para nuevas tablas que soporten la definición de conectores:
*   **`connectors`**: Definición base (Nombre, Tipo, Base URL, Esquema de Auth, Políticas de Retry por defecto).
*   **`connector_environments`**: Configuraciones específicas por ambiente (Dev/Prod) para credenciales y endpoints específicos.
*   **`connector_templates`**: Plantillas reutilizables para servicios comunes (PDF, Email, etc.).

### 1.2 API de Gestión (Node.js)
Implementar endpoints CRUD en el servicio `RA` para administrar el catálogo:
*   `POST /connectors`: Registrar nuevos servicios externos.
*   `PUT /connectors/{id}/config`: Gestionar credenciales de forma segura.
*   **Validación**: Implementar validación de esquemas JSON para asegurar que las configuraciones cumplan con los formatos esperados.

## Fase 2: Motor de Ejecución Robusto (Servicio Worker)
Actualizar el motor en Go para soportar la lógica avanzada de conectores y ejecución asíncrona.

### 2.1 Ejecutor de Conectores
Mejorar el manejo de `Tool` para interpretar definiciones de conectores:
*   **Cliente HTTP Resiliente**: Implementar un cliente interno que soporte `Retries` (reintentos con backoff exponencial) y `Timeouts` configurables por paso.
*   **Inyección de Credenciales**: Lógica segura para inyectar headers/tokens en tiempo de ejecución sin exponerlos en los logs.

### 2.2 Soporte Asíncrono (Webhooks)
Habilitar flujos de larga duración:
*   **Estado `Suspended`**: Permitir que una herramienta retorne un estado de "Espera" en lugar de "Éxito/Fallo".
*   **Persistencia de Estado**: Asegurar que el estado del Job se guarde correctamente mientras espera un evento externo.

## Fase 3: API Gateway y Orquestación
Unificar la entrada al sistema y manejar la lógica de "hidratación" de flujos.

### 3.1 Hidratación de Flujos
Modificar la lógica de inicio de flujo (en `RA` o un nuevo controlador):
*   Al iniciar una instancia, combinar la **Definición del Flujo** con la **Configuración del Conector** (resolviendo credenciales del ambiente actual) antes de enviarlo al Worker.

### 3.2 Endpoints de Ejecución y Callbacks
*   `POST /api/v1/execution/trigger`: Endpoint unificado para iniciar flujos (Modo Auto/API).
*   `POST /api/v1/callbacks/{execution_id}/{step_id}`: Endpoint para recibir webhooks de sistemas externos y reanudar flujos suspendidos.

## Resumen Técnico
1.  **RA (Node.js)**: Se convierte en el **Gestor de Catálogo y Configuración**.
2.  **Worker (Go)**: Se convierte en el **Ejecutor Resiliente** con soporte para pausas y reintentos.
3.  **Base de Datos**: Se expande para alojar definiciones ricas de conectores.
